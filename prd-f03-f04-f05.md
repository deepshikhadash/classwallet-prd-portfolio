# F-03 · Program Health Dashboard
## ClassWallet Feature PRD
**Author:** Deepshikha Dash, PM · [github.com/Deepshikhadash](https://github.com/Deepshikhadash)  
**Feature ID:** CW-F-03 · **Status:** Draft v1.0 · **Priority:** P0 — Q3 2026  
**Stakeholders:** Agency Administrators · Program Directors · Engineering · Design

---

## 1. Problem Statement

Admins currently have no single view of program health. Spend data, compliance flags, and recipient status live in separate views requiring multiple navigation steps to assemble a picture of how a program is running. During high-stakes periods — program deadlines, audit preparation, legislative reporting windows — this fragmentation is operationally dangerous.

**An admin who can't see the full picture in 30 seconds cannot intervene in time.**

> **PM Note:** At DHS/USCIS I built and consumed compliance dashboards for federal data products. The most common failure mode wasn't bad data — it was data scattered across tools. Consolidation into a single health view is the highest-leverage product investment for an admin-facing compliance platform.

---

## 2. Goals & Success Metrics

| Metric | Target |
|---|---|
| Time-to-program-overview from login | ≤ 30 seconds |
| Manual data reconciliation effort | ↓ 60% |
| Dashboard weekly active admin rate | ≥ 70% |
| Dashboard load time (P95, 10K recipients) | ≤ 3 seconds |
| Admin satisfaction with dashboard (CSAT) | ≥ 4.2 / 5.0 |

---

## 3. User Stories & Acceptance Criteria

---

### Story CW-F03-S1 — Program-Level KPI Summary

**As an** agency administrator,  
**I want to** see a program-level KPI summary on the dashboard home screen,  
**So that** I can assess program health in under 30 seconds without navigating multiple views.

**Acceptance Criteria:**
- [ ] Dashboard displays on login if admin has an active program — not buried behind navigation
- [ ] KPI cards show: Total Allocated ($), Total Spent ($), Total Remaining ($), % Utilization, Active Recipients, Flagged Transactions (count)
- [ ] Each KPI card is color-coded: 🟢 green (on track), 🟡 amber (attention needed), 🔴 red (intervention required) — thresholds configurable per program
- [ ] Admin can switch between programs from a top-level selector if they manage multiple
- [ ] Data freshness indicator shows time of last update — auto-refreshes every 15 minutes
- [ ] Manual refresh button triggers update within 60 seconds
- [ ] Dashboard fully functional on 1280px desktop and 768px tablet

**Definition of Done:** ✅ Coded · ✅ QA tested · ✅ Color threshold logic verified · ✅ Multi-program selector tested · ✅ Refresh timing verified · ✅ Responsive layout tested · ✅ Release notes updated

---

### Story CW-F03-S2 — Drill-Down by Recipient Cohort

**As an** agency administrator,  
**I want to** drill down into any KPI to see recipient-level detail,  
**So that** I can identify which specific recipients are driving a flag or trend and take targeted action.

**Acceptance Criteria:**
- [ ] Clicking any KPI card opens a recipient-level list filtered to that metric
- [ ] List shows: recipient ID/name, relevant metric value, last transaction date, account status
- [ ] Admin can sort by any column; filter by district, school, or date range
- [ ] Admin can take actions directly from the list: send notification, flag for review, export selected rows
- [ ] Drill-down list paginated at 50 recipients per page; exports up to 10,000 rows
- [ ] Back navigation returns to dashboard without losing filter state

**Definition of Done:** ✅ Coded · ✅ QA tested · ✅ Sort/filter tested · ✅ Export verified (10K row limit) · ✅ State persistence on back-nav tested · ✅ Release notes updated

# F-04 · State Reporting Automation
## ClassWallet Feature PRD
**Author:** Deepshikha Dash, PM · [github.com/Deepshikhadash](https://github.com/Deepshikhadash)  
**Feature ID:** CW-F-04 · **Status:** Draft v1.0 · **Priority:** P0 — Q3 2026  
**Stakeholders:** Agency Administrators · Compliance/Legal · CSM · Engineering

---

## 1. Problem Statement

State agencies must submit compliance reports to oversight bodies on regular cycles. Currently, generating these reports requires admins to export raw data from ClassWallet, manually reformat it to match each state's required template, and reconcile discrepancies — a process taking **2–3 hours per report cycle**.

With ClassWallet operating across 33 states and multiple programs per state, this is a massive recurring cost, a significant error source, and a barrier to renewal. Solving it creates a meaningful competitive moat: no other platform in this space offers state-formatted automated reporting.

> **PM Note:** I built compliance reporting frameworks at both DHS/USCIS and EDC. The 2–3 hour reconciliation problem is endemic across public program administration. This is a solvable product problem with high strategic value — agencies that experience this feature won't leave for a competitor that doesn't have it.

---

## 2. Goals & Success Metrics

| Metric | Baseline | Target |
|---|---|---|
| Report generation time | 2–3 hours | ≤ 5 minutes |
| Report formatting errors | Manual baseline TBD | ↓ 90% |
| Admin report generation NPS | Unknown | ≥ 50 |
| State templates configured at launch | 0 | ≥ 10 states |
| Admin time saved per quarter | TBD | ≥ 6 hours per admin |

---

## 3. User Stories & Acceptance Criteria

---

### Story CW-F04-S1 — One-Click Compliance Report

**As an** agency administrator,  
**I want to** generate a state-formatted compliance report with one click,  
**So that** I can meet my reporting obligations in minutes instead of hours.

**Acceptance Criteria:**
- [ ] Admin selects: report type, program, and date range — **3 inputs maximum**
- [ ] Report generates in ≤ 5 minutes for programs with up to 10,000 recipients and 12 months of data
- [ ] Progress indicator shown during generation; admin can navigate away and return
- [ ] Output formats: PDF (formatted for state submission) and CSV (raw data) — both available from same screen
- [ ] PDF output matches the selected state's official reporting template exactly — column order, field labels, and summary calculations
- [ ] Report includes a data certification block with admin name, date, and program ID
- [ ] Generated report saved to report history: filename, date, admin ID, program, report type
- [ ] Admin receives email notification when report is ready (for reports taking > 60 seconds)
- [ ] Report data accurate to within 15 minutes of export timestamp — freshness indicator shown

**Definition of Done:** ✅ Coded · ✅ QA tested · ✅ PDF template verified against state format · ✅ CSV column structure verified · ✅ Performance tested (5-min threshold) · ✅ Email notification tested · ✅ Report history persistence verified · ✅ Release notes updated

**So that** new state programs can be onboarded to automated reporting quickly.

- [ ] Template builder supports: field mapping, custom calculation rules, header/footer text, logo upload
- [ ] Template can be previewed with sample data before activation
- [ ] Templates are versioned — previous versions retained for audit purposes
- [ ] New template goes live within 24 hours of configuration — no code deployment required

# F-05 · AI Spend Anomaly Detection
## ClassWallet Feature PRD
**Author:** Deepshikha Dash, PM · [github.com/Deepshikhadash](https://github.com/Deepshikhadash)  
**Feature ID:** CW-F-05 · **Status:** Draft v1.0 · **Priority:** P1 — Q4 2026  
**Stakeholders:** Agency Administrators · Compliance/Legal · Engineering · AI/ML Team

---

## 1. Problem Statement

ClassWallet's existing eligibility enforcement blocks clearly ineligible transactions at the rule level. But it cannot detect patterns that are technically eligible yet anomalous: a recipient who makes 15 small purchases in one day, whose spend velocity spikes 10x in a week, or whose category usage shifts suddenly. These patterns may indicate account compromise, fraud, or misuse that passes individual transaction checks but fails at the pattern level.

**Rule-based systems catch known violations. AI catches unknown patterns.**

> **PM Note:** My Product School AI PM Certificate and AI Evals & Agents Certificate directly inform this PRD. I've scoped this feature with the four compliance AI gates I apply to every AI product decision. This feature does not ship without all four gates passing — that's not a preference, it's a PM principle for any AI system touching public funds.

---

## 2. Compliance AI Gates — Non-Negotiable

These four gates must pass before any production deployment. They are requirements, not guidelines.

| Gate | Requirement | Why It Matters |
|---|---|---|
| **1. Accuracy Threshold** | Model precision ≥ 95% on held-out historical data before deploy | Low false positives — admins cannot afford to chase phantom flags |
| **2. Explainability** | Every flag includes a plain-language reason (e.g., "15 transactions in 24 hours vs. recipient avg of 2/week") | Black-box decisions are not defensible in publicly funded programs |
| **3. Bias Audit** | Flag rate variance across recipient demographic groups ≤ 10% | Disparate flagging would discriminate against recipients by income, race, or geography |
| **4. Human Override** | All AI flags require admin review before any account action — no automated suspensions | Irreversible automated actions on financial accounts are unacceptable |

---

## 3. Goals & Success Metrics

| Metric | Target |
|---|---|
| Anomaly detection recall on confirmed cases | ≥ 70% |
| False positive rate (precision) | ≤ 5% (precision ≥ 95%) |
| Admin review time per flag | ≤ 3 minutes (explainability requirement) |
| Demographic flag rate variance | ≤ 10% across groups |
| Model uptime / availability | ≥ 99.5% |

---

## 4. User Stories & Acceptance Criteria

---

### Story CW-F05-S1 — Anomaly Flag in Admin Alert Center

**As an** agency administrator,  
**I want to** see AI-generated anomaly flags for unusual recipient spend patterns in my alert queue,  
**So that** I can investigate potential misuse before it escalates to an audit finding.

**Acceptance Criteria:**
- [ ] Anomaly flags appear in the existing Alert Center (F-01) with type label "AI Anomaly"
- [ ] Each flag displays: recipient ID, anomaly type, plain-language explanation, severity (High/Medium), and date detected
- [ ] Plain-language explanation **required** — e.g., "14 transactions in 24 hours (recipient avg: 1.8/week)"
- [ ] Admin can click flag to view full transaction detail for the flagged period
- [ ] Admin resolves flag with one of three dispositions: "Reviewed — No Action", "Reviewed — Contacted Recipient", "Reviewed — Escalated to Compliance"
- [ ] All review actions logged to audit trail with timestamp and admin ID
- [ ] **AI flags do NOT automatically restrict recipient account** — human decision required for any account action
- [ ] Flags older than 30 days without resolution surface in a "Stale Flags" view for admin follow-up

**Definition of Done:** ✅ Coded · ✅ QA tested · ✅ Plain-language explanation format verified · ✅ Human-override logic verified (no auto-restrict) · ✅ Audit log confirmed · ✅ Stale flag surfacing tested · ✅ Release notes updated

---

### Story CW-F05-S2 — Model Performance Dashboard (Internal)

**As a** ClassWallet ML/compliance team member,  
**I want to** monitor the anomaly detection model's precision, recall, and demographic parity in real time,  
**So that** I can catch model drift or bias before it affects recipients.

**Acceptance Criteria:**
- [ ] Internal dashboard (not visible to agency admins) shows: daily precision, recall, F1 score, and flag volume
- [ ] Bias monitor shows flag rate by: state, program type, and recipient income quartile
- [ ] Automated alert triggers if precision drops below 90% OR demographic flag rate variance exceeds 15%
- [ ] Model versioning: each deployed version logged with training date, accuracy metrics, and bias audit results
- [ ] Rollback capability: previous model version can be restored within 4 hours of alert

**Definition of Done:** ✅ Coded · ✅ QA tested · ✅ Alert thresholds verified · ✅ Rollback process tested · ✅ Version logging confirmed · ✅ Access control (internal only) verified
